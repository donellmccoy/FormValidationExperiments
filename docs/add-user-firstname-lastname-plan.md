# Plan: Add `FirstName` and `LastName` to ASP.NET Identity

**Scope (Option 1):** model + migration only. No API endpoints, UI, or claims changes.

## Files to change

### 1. [ECTSystem.Persistence/Models/ApplicationUser.cs](../ECTSystem.Persistence/Models/ApplicationUser.cs)

Add two nullable string properties:

```csharp
using System.ComponentModel.DataAnnotations;
using Microsoft.AspNetCore.Identity;

namespace ECTSystem.Persistence.Models;

public class ApplicationUser : IdentityUser
{
    [PersonalData]
    [MaxLength(100)]
    public string? FirstName { get; set; }

    [PersonalData]
    [MaxLength(100)]
    public string? LastName { get; set; }
}
```

- `[PersonalData]` flags them for Identity's GDPR download/delete tooling.
- `[MaxLength(100)]` so the migration emits `nvarchar(100)` instead of `nvarchar(max)`.
- Nullable so existing rows stay valid without a default.

## Out of scope (explicitly deferred)

- Custom `/register` endpoint (built-in `MapIdentityApi<ApplicationUser>()` `RegisterRequest` is sealed and won't accept the new fields).
- `Register.razor` form fields, `RegisterRequest` DTO, `AuthService.RegisterAsync` signature.
- `IUserClaimsPrincipalFactory<ApplicationUser>` to emit `GivenName` / `Surname` claims in the JWT.
- Backfill of existing user rows.

After this lands, new users created via `UserManager` directly (e.g., a seeder) can set the fields, but the public `/register` endpoint will continue to leave them `NULL` until the "full vertical" pieces above are added.
