# PostgreSQL Alternative Persistence Layer — Migration Plan

## Executive Summary

Add PostgreSQL as an alternative database provider to the ECTSystem application alongside the existing SQL Server provider. The app will support **runtime provider switching** via configuration (`appsettings.json`), with separate EF Core migration assemblies per provider.

---

## Current State Analysis

### SQL Server–Specific Features in Use

| Feature | Location | PostgreSQL Equivalent |
|---------|----------|----------------------|

## Architecture Decision: Provider Switching Strategy

### Recommended: Configuration-based provider switching (single Persistence project)

```
appsettings.json
{
  "DatabaseProvider": "SqlServer",    // or "PostgreSQL"
  "ConnectionStrings": {
    "EctDatabase": "Server=localhost;Database=ECT;...",
    "EctDatabasePg": "Host=localhost;Database=ect;Username=ect_user;Password=..."
  }
}
```

**Why this approach:**
- Minimal code duplication — entity configurations shared between providers
- Single `EctDbContext` class — provider-agnostic
- Provider-specific SQL (e.g., `HasDefaultValueSql`, `HasFilter`) handled via a small abstraction
- Separate migration assemblies keep migration history clean per provider

**Alternative rejected:** Separate `ECTSystem.Persistence.PostgreSql` project — too much duplication for the small number of provider-specific differences.

#### 2b. Create SQL dialect helper

**New file:** `ECTSystem.Persistence/Data/SqlDialect.cs`

Encapsulates the small number of provider-specific SQL expressions:

```csharp
namespace ECTSystem.Persistence.Data;

public interface ISqlDialect
{
    string UtcNow { get; }               // GETUTCDATE() vs now() AT TIME ZONE 'UTC'
    string BoolFalse { get; }            // 0 vs false
    string QuoteIdentifier(string id);   // [Col] vs "Col"
}

public sealed class SqlServerDialect : ISqlDialect
{
    public string UtcNow => "GETUTCDATE()";
    public string BoolFalse => "0";
    public string QuoteIdentifier(string id) => $"[{id}]";
}

public sealed class PostgreSqlDialect : ISqlDialect
{
    public string UtcNow => "now() AT TIME ZONE 'UTC'";
    public string BoolFalse => "false";
    public string QuoteIdentifier(string id) => $"\"{id}\"";
}
```

#### 2c. Thread dialect through entity configurations

Entity configurations that use `HasDefaultValueSql` or `HasFilter` with raw SQL need the dialect. Approaches (pick one):

**Option A — Static property on DbContext (simplest):**

```csharp
// EctDbContext.cs
public static ISqlDialect Dialect { get; set; } = new SqlServerDialect();
```

Set during DI registration before `ApplyConfigurationsFromAssembly`. Configurations read `EctDbContext.Dialect`.

**Option B — Constructor injection into configurations:**

Pass `ISqlDialect` to configurations via `ConfigureConventions` or a custom `IModelCustomizer`. More complex but more testable.

**Recommendation:** Option A — only 3 configuration files use raw SQL, and this avoids overengineering.

### Phase 3: Concurrency Token Strategy (RowVersion)

This is the **highest-risk change** because SQL Server `rowversion` is a database-managed 8-byte auto-incrementing value with no direct PostgreSQL equivalent.

#### Options

| Strategy | Pros | Cons |
|----------|------|------|
| **A. `xmin` system column** | Zero schema change; Npgsql has built-in support via `UseXminAsConcurrencyToken()` | Only detects row-level changes; reset on VACUUM FULL; not portable between providers |
| **B. Manual `uint Version` column** | Portable; explicit; works on both providers | Requires model change (`byte[] → uint`); must update all OData Delta.Patch calls to increment version |
| **C. Conditional configuration** | Keep `byte[] RowVersion` for SQL Server; use `xmin` for PostgreSQL | Two code paths; more complex testing |

#### Recommendation: Option C — Conditional configuration

Keep the existing `AuditableEntity.RowVersion` property and `[Timestamp]` attribute for SQL Server. For PostgreSQL, configure `xmin` as the concurrency token and ignore the `RowVersion` property:

```csharp
// In EctDbContext.OnModelCreating, after ApplyConfigurationsFromAssembly:
if (Database.IsNpgsql())
{
    foreach (var entityType in modelBuilder.Model.GetEntityTypes())
    {
        if (typeof(AuditableEntity).IsAssignableFrom(entityType.ClrType))
        {
            // Ignore the byte[] RowVersion property — PostgreSQL doesn't have rowversion
            modelBuilder.Entity(entityType.ClrType).Ignore(nameof(AuditableEntity.RowVersion));

            // Use PostgreSQL xmin system column as concurrency token
            modelBuilder.Entity(entityType.ClrType).UseXminAsConcurrencyToken();
        }
    }
}
```

**Impact on OData/API:** The `RowVersion` property will not appear in OData responses for PostgreSQL. OData concurrency relies on ETags — EF Core's `xmin`-based concurrency will handle conflicts at the database level transparently. The API's `AuditSaveChangesInterceptor` and Delta.Patch calls do not manually read/write RowVersion, so no API changes needed.

### Phase 4: Update Entity Configurations

#### 4a. `HasDefaultValueSql` — 3 configurations

**Files:** `WorkflowModuleConfiguration.cs`, `WorkflowStateLookupConfiguration.cs`, `WorkflowTypeConfiguration.cs`

Change from:
```csharp
builder.Property(e => e.CreatedDate).HasDefaultValueSql("GETUTCDATE()");
```
To:
```csharp
builder.Property(e => e.CreatedDate).HasDefaultValueSql(EctDbContext.Dialect.UtcNow);
```

#### 4b. `HasFilter` — 1 explicit configuration + snapshot-level filters

**File:** `LineOfDutyCaseConfiguration.cs`

Change from:
```csharp
builder.HasIndex(e => e.IsDeleted)
       .HasDatabaseName("IX_Cases_IsDeleted")
       .HasFilter("[IsDeleted] = 0");
```
To:
```csharp
var d = EctDbContext.Dialect;
builder.HasIndex(e => e.IsDeleted)
       .HasDatabaseName("IX_Cases_IsDeleted")
       .HasFilter($"{d.QuoteIdentifier("IsDeleted")} = {d.BoolFalse}");
```

> **Note:** The unique index filters for `CaseId IS NOT NULL`, `UserId IS NOT NULL`, `INCAPId IS NOT NULL`, and `MEDCONId IS NOT NULL` are generated **automatically** by EF Core for unique indexes on nullable columns when using SQL Server. With Npgsql, EF Core will generate the correct PostgreSQL syntax automatically — these are **not in the entity configuration code** (they only appear in migration Designer files/snapshots). No manual changes needed for these.

### Phase 5: Update DI Registration

**File:** `ECTSystem.Api/Extensions/ServiceCollectionExtensions.cs`

```csharp
public static IServiceCollection AddDatabase(
    this IServiceCollection services,
    IConfiguration configuration)
{
    var provider = configuration.GetValue<string>("DatabaseProvider") ?? "SqlServer";

    // Set dialect for entity configurations
    EctDbContext.Dialect = provider == "PostgreSQL"
        ? new PostgreSqlDialect()
        : new SqlServerDialect();

    services.AddPooledDbContextFactory<EctDbContext>(options =>
    {
        if (provider == "PostgreSQL")
        {
            options.UseNpgsql(
                configuration.GetConnectionString("EctDatabasePg"),
                npgsql =>
                {
                    npgsql.UseQuerySplittingBehavior(QuerySplittingBehavior.SplitQuery);
                    npgsql.EnableRetryOnFailure(
                        maxRetryCount: 5,
                        maxRetryDelay: TimeSpan.FromSeconds(30),
                        errorCodesToAdd: null);
                    npgsql.MigrationsAssembly("ECTSystem.Persistence.Migrations.PostgreSql");
                });
        }
        else
        {
            options.UseSqlServer(
                configuration.GetConnectionString("EctDatabase"),
                sql =>
                {
                    sql.UseQuerySplittingBehavior(QuerySplittingBehavior.SplitQuery);
                    sql.EnableRetryOnFailure(
                        maxRetryCount: 5,
                        maxRetryDelay: TimeSpan.FromSeconds(30),
                        errorCodesToAdd: null);
                    // SQL Server migrations stay in default assembly
                });
        }
    }, poolSize: 32);

    // Same pattern for EctIdentityDbContext...
    services.AddPooledDbContextFactory<EctIdentityDbContext>(options =>
    {
        if (provider == "PostgreSQL")
        {
            options.UseNpgsql(
                configuration.GetConnectionString("EctDatabasePg"),
                npgsql => npgsql.MigrationsAssembly("ECTSystem.Persistence.Migrations.PostgreSql"));
        }
        else
        {
            options.UseSqlServer(
                configuration.GetConnectionString("EctDatabase"));
        }
    });

    // Rest of registration unchanged...
}
```

namespace ECTSystem.Persistence.Migrations.PostgreSql;

public class EctDbContextDesignTimeFactory : IDesignTimeDbContextFactory<EctDbContext>

### Phase 8: DateTime/UTC Handling

**Current state:** `EctDbContext.OnModelCreating` applies `ValueConverter<DateTime, DateTime>` to normalize all DateTime values to UTC. This works with SQL Server's `datetime2` (which stores no timezone info).

**PostgreSQL behavior:** Npgsql maps `DateTime` to `timestamp with time zone` by default and **requires** `DateTimeKind.Utc` on all values. The existing UTC ValueConverters are **compatible** — they ensure all values are UTC before storage, which is exactly what Npgsql requires.

**Action:** No changes needed. The existing UTC converters will work correctly with PostgreSQL. The Npgsql provider will store them as `timestamptz`.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `xmin` concurrency resets on VACUUM FULL | Low | Medium | `xmin` is stable for normal operations; VACUUM FULL is rare and typically admin-scheduled |
| DateTime UTC converter incompatibility with Npgsql | Low | High | Already validated — existing converters produce `DateTimeKind.Utc`, which Npgsql requires |
| OData Delta.Patch fails with missing RowVersion on PostgreSQL | Low | Medium | `RowVersion` property is `Ignore()`d; EF Core xmin concurrency is transparent |
| HasFilter SQL syntax breaks on wrong provider | Medium | Medium | Dialect abstraction ensures correct quoting per provider |
| Seed data `RowVersion = new byte[0]` causes PostgreSQL error | Low | Low | `RowVersion` is ignored for PostgreSQL entities |
| Existing SQL Server migrations break | None | High | SQL Server migrations are untouched; PostgreSQL gets its own assembly |
